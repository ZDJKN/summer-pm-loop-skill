# 26.05 复盘沉淀 🔁

> 让错误有价值、能力可复制。**忌**只经历不沉淀、凭感觉做事、问题爆发才救火；**记**随时反思/复盘/沉淀，让项目运转不依赖任何特定个人，而依赖标准体系。
> 上级 → [[26 项目管理PM]] ｜ 信源：《PM工作标准》运行过程·复盘沉淀 + 救火Step12
> 🟢 Phase 1 已原子化

---

## 总纲 · 复盘是飞轮

复盘是研究框架 **D→A 的回流**：真实复盘 → 变成新信源 → 再蒸馏进知识层 → 迭代 skill。越用越准（见 [[26.00 研究框架与路线图]]）。

事后阶段定位：**有闭环、可复制**。关键目标=经验资产化、能力可复制。管理重点=数据归因、流程迭代、资产入库。

---

## F1. 复盘的三个时机

- **场景**：不是只在项目结束才复盘。
- **✅标准动作**：

| 时机 | 触发 | 做什么 |
|---|---|---|
| 随时心流复盘 | 解决完关键问题立刻 | 趁记忆还热，想哪里好/哪里不好。结论必含"下次遇到类似第一步做什么" |
| 节点复盘 | 返工>2次叫停时 | review 找因**不找背锅人**（见 [[26.02 运行管理]] A5） |
| 项目复盘 | 项目结束 | 数据归因 + 流程迭代 + 资产入库 |

- **❌误区**：复盘=等项目结束才做（错，要随时心流复盘）。
- **📌留痕**：复盘条目 / 项目复盘报告。

## F2. 心流复盘两问

- **场景**：解决完一个关键问题/节点后立刻。
- **✅标准动作**：

| 情况 | 复盘内容 |
|---|---|
| 满意度/数据**提升** | 做对了什么？哪些可复用？能沉淀进SOP吗？ |
| 满意度/数据**下降** | 哪做错了？方向？目标？需求把控？ |

- **🚨预警**：同样的问题在下一个项目里又发生 = 复盘没沉淀进SOP。
- **📌留痕**：随手复盘记录。

## F3. 资产化（让经验不停在某人脑子里）

- **场景**：复盘产出要进「组织形式」的库。
- **✅标准动作（三类资产）**：
  - **案例库** → `_复盘案例/`，含**客户画像**：偏好 / 雷区 / 历史争议点 / 偏好的反馈方式 / 对接节奏。
  - **SOP迭代** → 回写到 26.01~26.04 对应知识点，标注版本（别破坏原子笔记结构，新增"迭代记录"段）。
  - **模板沉淀** → `_模板/`（checklist/话术/表格）。
- **❌误区**：资料卡在某人手上、客户历史沟通散在私人聊天框。
- **📌留痕**：案例库条目 + SOP版本记录。
- 🔗 配合 [[26.02 运行管理]] B3 人力备案的资料库要求。
